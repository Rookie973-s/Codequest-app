Chapter 1 · Level 1.1

Welcome to the World of Programming

Quick Guide

Programming means writing a to-do list so clear and complete that a computer can follow it without guessing — and it's how every app, website, and game gets made.

Before you touch a single line of Python, it helps to slow down and ask a bigger question: what is programming, actually? Programming means giving a computer instructions so exact that it can follow them with zero common sense of its own. "Coding" is the everyday word for the act of typing those instructions in a language a computer understands, while "programming" is the bigger skill of planning, designing, and solving the problem behind the code — coding is one part of programming, not the whole thing. A computer program is simply the finished list of instructions, saved and ready to run any time. Traffic lights, microwave timers, video game enemies, and the maps app on a phone are all running programs — someone, somewhere, wrote out exactly what should happen and when.

Why This Matters

  • Coding is typing the instructions; programming is the whole skill of planning and solving the problem.
  • Computers are fast, but they have zero imagination — they only do what you actually tell them.
  • Every app, game, and website you've ever used started life as a list of steps just like this.
Real-Life Examples
A traffic light cycling red-yellow-green, a washing machine's spin cycle, autocorrect on your phone, and every enemy in a video game are all programs — step-by-step instructions someone wrote in advance.

Chapter 4 · Level 1.4

Writing Your First Python Program

Quick Guide

print() is the first command every programmer learns — it makes the computer show words on the screen — and errors are not failures, they're clues.

Almost every programmer in history writes a "Hello, World!" program as their very first line of code, using print() to make the computer speak. Comments, written after a #, are notes for humans that Python quietly ignores — handy for explaining what your code does. When an error shows up, it isn't a failure — it's the computer politely asking you to double-check one specific line. The big three you'll meet constantly are SyntaxError (broken grammar, like a missing quote), NameError (a word Python doesn't recognize), and IndentationError (spacing that doesn't match what Python expects). An error message isn't a sign you failed; it's the computer politely pointing at one specific line and asking you to take a second look — that's the debugging mindset every programmer builds over time.

Why This Matters

  • print() lets you check that your code is really doing what you expect.
  • Errors always point to a specific line — read them like clues, not scary warnings.
  • Every professional programmer, every single day, still reads error messages like these.
Debug Detective
SyntaxError usually means a missing quote or colon. NameError means Python doesn't recognize a word you typed. IndentationError means your spacing doesn't match what Python expects.

Chapter 5 · Level 1.5

Variables & Data Types: Labeled Backpack Pockets

Quick Guide

A variable is a labeled storage box that holds a piece of information — like a name, a score, or whether the game is over — so you can use it again later.

A variable is a labeled pocket that stores a piece of information for later — think of a backpack with a pocket named player_name and another named score. Naming rules are simple: names can't start with a number, can't contain spaces, and are case-sensitive, so score and Score are different variables entirely. Every value also belongs to a data type: strings for text, integers for whole numbers, floats for decimals, and booleans for simple True/False switches like is_game_over. The built-in type() function tells you exactly which data type any value is. Giving your variables clear names, like player_health instead of just x, makes your own code much easier to understand when you come back to it.

Why This Matters

  • Clear variable names make code easier to read — for you and for teammates.
  • Every value has a type, and Python needs to know that type to use it correctly.
  • Booleans are the simplest and most powerful type of all — just True or False.
Did You Know?
Games use booleans constantly behind the scenes — has_key, is_jumping, and is_paused are always exactly True or False, nothing in between.

Chapter 6 · Level 1.6

Talking to the User

Quick Guide

input() lets your program ask the player a question and wait for their answer — but that answer always arrives as plain text, even if it looks like a number.

The input() function pauses your program to ask the player a question and wait for an answer — but it always hands that answer back as text, even when it looks like a number. Pairing print() with input() lets a program hold a real back-and-forth "conversation," and f-strings — written as f"Hello, {name}!" — let you drop a variable straight into a sentence without clunky joining. Wrapping the answer in int() or float(), called type conversion, is how you teach Python to actually do math with whatever the player typed, while str() goes the other direction, turning a number back into text. That's why converting the answer with int() or float() is such an important habit — it's the difference between Python reading "5" as a word versus a number it can actually add and subtract.

Why This Matters

  • Wrap input() in int() or float() before doing any math with the answer.
  • f-strings let you drop a variable straight into a sentence, like "Hello, {name}!"
  • This is the first moment your program truly listens and responds to the player.
Debug Detective
Seeing "TypeError: unsupported operand type(s) for +: 'int' and 'str'"? You forgot to convert input() text into a number first — the single most common beginner mistake.

Chapter 7 · Level 1.7

Python Can Do Math

Quick Guide

Python's arithmetic operators handle addition, subtraction, multiplication, division, and two special beginner favorites: floor division and modulus.

Python's core arithmetic operators are +, -, *, / (division, which always gives a decimal), // (floor division, which rounds down to a whole number), % (modulus, which returns only the remainder), and ** (exponents, for powers). Python follows the same order of operations you learned in math class — parentheses first, then exponents, then multiplication/division, then addition/subtraction. Putting all of this together into a simple calculator that takes two numbers with input(), converts them, and prints a result is a perfect first mini project. Modulus (%) feels new to most beginners, but it's exactly how games check things like "every 3rd frame" or "is this number even" — even_number % 2 == 0.

Why This Matters

  • // gives a whole number answer; % gives you only the leftover remainder.
  • Order of operations in Python works exactly like it does in math class.
  • Every score system, health bar, and timer in a game runs on this same simple math.
Did You Know?
Modulus (%) is secretly one of the most useful tools in game design — it's how developers make something happen "every Nth time" without any extra counting logic.

Chapter 9 · Level 2.1

Comparison Operators

Quick Guide

Comparison operators like ==, !=, >, and < let Python compare two values and decide if a statement is True or False.

Comparison operators are how a program answers a yes-or-no question, like "did the player press jump?" or "is health at zero?" Python compares two values with == (equal to), != (not equal to), >, <, >=, and <= — and every single comparison evaluates to exactly one of two values: True or False. This is the exact building block every decision in your game will be made from. Mixing up = (assigns a value) with == (compares two values) is the single most common beginner slip — Python will even throw an error to remind you.

Why This Matters

  • == compares two values; = assigns a value — mixing them up is a classic beginner error.
  • Every comparison in Python resolves to exactly True or False, nothing else.
  • These operators are the raw material every if statement is built from.

Chapter 10 · Level 2.2

Boolean Logic

Quick Guide

Boolean logic lets you combine multiple True/False questions into one bigger decision, using and, or, and not.

Real decisions are rarely just one question — a door might only open if the player has a key AND has enough energy. Python's logical operators combine booleans: and needs both sides to be True, or needs just one side to be True, and not simply flips True to False (and back). Reading these combinations out loud in plain English is the fastest way to check your logic is actually doing what you think it's doing. A door that opens with has_key and not is_locked reads almost exactly like the English sentence describing the rule.

Why This Matters

  • and requires every condition to be True; or only needs one.
  • not flips a boolean, turning True into False and False into True.
  • Combining conditions is how games handle real rules like locked doors or team victories.
Did You Know?
Boolean logic is named after George Boole, a 19th-century mathematician who worked all of this out over 100 years before the first computer existed.

Chapter 12 · Level 2.4

Loops: While & For

Quick Guide

Loops let your program repeat instructions automatically — while repeats until a condition changes, for repeats a set number of times.

Loops let your program repeat instructions automatically, instead of you copying and pasting the same code again and again. A while loop repeats as long as a condition stays True — perfect for "keep playing until the player quits." A for loop repeats a set number of times, often over range() — perfect for "roll the dice 5 times." Inside either loop, break exits immediately, and continue skips straight to the next repeat. Every game loop you'll build in Week 4 is really just a while loop that never intends to stop until the player says so.

Why This Matters

  • Use while when you don't know how many times you'll repeat; use for when you do.
  • break stops a loop immediately; continue skips to the next round.
  • A while loop with no way to become False runs forever — this is called an infinite loop.
Debug Detective
Program frozen and not responding? You likely wrote an infinite loop — a while condition that never has the chance to become False.

Chapters 15–17 · Level 3.1

Functions, Parameters & Return Values

Quick Guide

A function is a reusable recipe for a task — write the steps once with def, feed it ingredients with parameters, and get a finished result back with return.

A function is a named, reusable recipe for a block of code — write it once with def, then call it by name anywhere you need that exact task done again instead of copy-pasting. Parameters are the ingredients you hand the function each time you call it, written inside the parentheses, like def heal(amount):. And return is how the function hands you back a finished result, so you can store it in a variable and use it elsewhere in your program. Parameters let a single function handle many different situations — heal(10) and heal(50) run the exact same recipe with different ingredients.

Why This Matters

  • def starts a function; return sends back a result you can actually use.
  • Parameters let one function handle many different situations.
  • Huge games — even Minecraft — are organized out of small, well-named functions working together.
Debug Detective
A function that print()s an answer but doesn't return it will hand back None if you try to store its result in a variable — return and print are not the same thing.
defparametersreturn

Chapters 18–19 · Level 3.2

Lists & Dictionaries: Organizing Game Data

Quick Guide

A list stores items in order, starting at position 0; a dictionary stores items by a name you choose instead of a position.

A list, written like inventory = ["sword", "shield", "potion"], stores items in order, starting at position 0 — so inventory[0] is "sword." A dictionary, written like player = {"name": "Alex", "hp": 100}, stores items by a name you choose, called a key, instead of a position — so player["hp"] instantly gets 100. Together, lists and dictionaries are how real games organize everything from a player's inventory to an entire level's enemy data. Reach for a list when order matters (a queue of enemies), and a dictionary when named lookup matters (one player's stats).

Why This Matters

  • Lists keep order and are accessed by position, starting at 0.
  • Dictionaries are accessed by a key name instead of a position.
  • Almost every piece of game data — inventories, stats, high scores — lives in one of these two structures.
Did You Know?
Python lists start counting at 0, not 1 — so a 3-item list's last item is at position 2, a detail that trips up nearly every new programmer at least once.

Chapters 28–30 · Level 4.3

Score, Lives & Timers

Quick Guide

A visible score, a lives counter, and an on-screen timer are what turn a moving shape into a game you can actually win or lose.

A real game needs a visible score, and the trick in turtle is a second, invisible pen-turtle whose only job is writing text near the top of the screen with pen.write() — clearing it with pen.clear() before every rewrite is essential, or old and new scores overlap into a mess. A lives system works the same way: a variable like lives = 3 counts down on every miss, and hitting zero ends the game. A timer is just another variable, counting up or down once per pass through the game loop, displayed the same way as score. Score, lives, and timers are really the same trick three times over: a variable, a rule for changing it, and pen.write() to show it.

Why This Matters

  • pen.write() displays text; pen.clear() must run first or old and new text overlap.
  • A lives variable that hits 0 is one of the simplest ways to end a game.
  • Visible feedback like a score, lives, or timer is what makes a game feel alive.
Debug Detective
Garbled, overlapping score text almost always means you forgot pen.clear() inside your update_score() function.

Chapters 32–33 · Level 5.1

Planning & Storyboarding Your Game

Quick Guide

Before writing any code, real developers storyboard their game on paper — sketching the idea, the goal, and how it's won or lost.

Before writing a single line, real developers storyboard: sketching the core loop, the win/lose condition, and the overall scope on paper first. A storyboard doesn't need to be fancy — a few boxes showing "start screen → gameplay → game over" is enough to catch problems before they cost you hours of code. Keeping scope small and finishable beats an ambitious idea that never gets done. Keeping your first idea small and finishable beats an ambitious one that never actually gets built.

Why This Matters

  • A small, finishable idea beats a giant idea that never gets done.
  • Sketching first saves you from rewriting a lot of code later.
  • Every professional game starts life as a rough sketch on paper.

Chapters 34–36 · Level 5.2

Writing, Testing & Debugging Your Code

Quick Guide

Build your most important feature first, test it constantly as you go, and when something breaks, hunt the bug down calmly, like a detective.

Build your core mechanic first, then layer features on top — writing the code for your storyboard one small, testable piece at a time. Testing means running your game after every meaningful change, not just once at the end, so you always know exactly which change caused a new problem. When you get stuck debugging, isolate the problem by commenting out code until it runs again, or explain your logic out loud — "rubber duck" debugging — and always save a working backup before trying anything risky. Commenting out sections of code until the program runs again is one of the fastest ways to narrow down exactly where a bug is hiding.

Why This Matters

  • Testing after every small change makes it obvious which change caused a bug.
  • "Rubber duck" debugging — explaining your code out loud — often reveals the fix on its own.
  • Always save a working backup before trying something risky.
Did You Know?
"Rubber duck debugging" gets its name from a programmer who kept a literal rubber duck on his desk to explain code to, line by line, until the bug became obvious.
